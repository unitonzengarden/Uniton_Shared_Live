=== CLIENT-SIDE SCHEMA HANDLING ===

types/taoV3.ts uses LOOSE typing for JSONB:
  - chart_object: Record<string, unknown>
  - root_payload: Record<string, unknown> | null

Hooks extract with type guards:
  - useBaziPillars: isEntaBaziRecord(chart) ? structured : isBaziChartRow(chart) ? JSONB : null
  - useBaziLuckPillars: payload.luck_pillars OR payload.luckPillars OR payload.dai_van
  - useBaziUsefulGod: payload.useful_god OR payload.usefulGod OR payload.dung_than
  - useZiweiPalaces: chart_object.palaces OR chart_object.cung

=== RATIONALE ===
- DB has no schema constraint enforcement on JSONB
- Multiple chart versions may have different shapes
- Lightweight client-side validation (try multiple field names) is more resilient
- KL-045 lesson: prefer JSONB read over migration for schema flexibility
